## 1. アプリケーションの全体像

`ssh-gete` は、**ローカルで動く AI エージェント（Claude Desktop / Codex / その他 MCP クライアント）に対して、SSH 越しのコマンド実行を「人間の承認付き」で安全に提供する** デスクトップアプリケーションです。

ひとことで言えば「**AI エージェント用の SSH 実行ゲートウェイ兼承認コンソール**」です。`gete` は「ゲート（gate）」を意識した名前で、エージェントと実機の間に立つ門番（承認関門）の役割を担います。

アプリは大きく三層で構成されます。

```
┌──────────────────────────────────────────────────────────────┐
│  デスクトップ GUI（Wails + AdminLTE/Bootstrap, バニラ JS）   │  ← 人間（管理者）が操作
│  ・接続先管理  ・承認キュー  ・MCP待受設定                    │
└───────────────▲──────────────────────────────────────────────┘
                │ Wails バインディング（Go の公開メソッドを JS から直接呼ぶ）
┌───────────────┴──────────────────────────────────────────────┐
│  Go バックエンド（App 構造体）                               │
│  ・接続先 / 承認キュー / エージェントポリシーの状態管理       │
│  ・SQLite への永続化                                          │
│  ・SSH 実行                                                   │
│  ・MCP（JSON-RPC over HTTP）サーバーの起動・制御              │
└───────────────▲──────────────────────────────────────────────┘
                │ JSON-RPC 2.0 over HTTP（Streamable HTTP, /mcp）
┌───────────────┴──────────────────────────────────────────────┐
│  MCP クライアント（AI エージェント）                          │  ← 機械が接続
│  ・list_hosts / execute_command /                            │
│    request_command_execution / get_command_history           │
└──────────────────────────────────────────────────────────────┘
```

ポイントは「**エージェントは直接 SSH を実行できない**」という点です。エージェントが `execute_command` を呼んでも、原則として実行ではなく **承認キューへの登録** で止まります。実際の SSH 実行は、人間が GUI 上で「承認して実行」を押したときか、もしくは特定エージェントに対し明示的に「承認省略（bypass）」を許可したときに限られます。この承認関門が本アプリの核心的な価値です。

## 2. 何を解決するアプリなのか（背景とコンセプト）

### 2.1 解決したい課題

AI エージェントにサーバー操作を任せたいが、`rm -rf` や `systemctl stop` のような破壊的コマンドを無断で実行されては困る——これが出発点です。一般的な MCP の SSH サーバーはコマンドをそのまま実行してしまうため、エージェントの判断ミスやプロンプトインジェクションが即座に本番事故になり得ます。

`ssh-gete` は、その間に「人間が見て承認する画面」を挟むことでリスクを抑えます。エージェントは「このホストでこのコマンドを、この理由で実行したい」と申請するだけで、判断と実行のトリガーは人間（または人間が事前に信頼を与えたエージェント）に委ねられます。

### 2.2 設計コンセプト

- **デフォルト安全（secure by default）**：MCP 経由のコマンドは、読み取り系（`whoami` のような Low リスク）であっても自動実行しない。すべて承認キューに入る。`AutoExecuteLowRisk` は UI からは常に false に固定される（`sanitizeMCPSettings` で強制）。
- **監査可能性**：すべてのコマンド要求は SQLite に記録され、削除しても接続先削除時には履歴を残す方針（「コマンド履歴は監査用に残します」）。
- **ローカル完結**：待受は既定で `127.0.0.1:8787`。外部公開を意図せず、ローカルのエージェントとだけ通信する前提。
- **段階的な信頼**：信頼できるエージェントには「承認省略」を付与でき、運用効率と安全のバランスを取れる。
- **資格情報の最小保持**：SSH 鍵パスフレーズは SQLite に保存せず、アプリ起動中のメモリ上にのみ保持する。

## 5. 起動からシャットダウンまでのライフサイクル

### 5.1 `main.go`

```go
//go:embed all:frontend/dist
var assets embed.FS

func main() {
    app := NewApp()
    err := wails.Run(&options.App{
        Title:     "ssh-gete",
        Width:     1440,
        Height:    960,
        MinWidth:  1180,
        MinHeight: 760,
        AssetServer: &assetserver.Options{Assets: assets},
        BackgroundColour: &options.RGBA{R: 238, G: 243, B: 249, A: 1},
        OnStartup:        app.startup,
        OnShutdown:       app.shutdown,
        Bind:             []interface{}{app},
    })
    ...
}
```

ここで起こることは4つです。

1. `//go:embed all:frontend/dist` により、ビルド済みフロントエンド（`frontend/dist`）をバイナリに埋め込む。配布物は単一実行ファイルで完結する。
2. `NewApp()` で `App` 構造体を生成（このとき SQLite を開き、既存データを読み込む）。
3. ウィンドウサイズ・タイトル・背景色を設定。
4. `OnStartup`/`OnShutdown`/`Bind` を登録して `wails.Run`。

### 5.2 `NewApp()`（`app.go`）

`NewApp` はアプリの初期状態を組み立てる関数で、順に次を行います（`app.go:134`）。

1. **DB パス決定**：`defaultDatabasePath()` → `<config>/ssh-gete/ssh-gete.sqlite`。`<config>` は `os.UserConfigDir()`（macOS なら `~/Library/Application Support`）か、環境変数 `SSH_GETE_CONFIG_DIR` で上書き可能。
2. **SQLite オープン**：`openStore(dbPath)`。失敗してもログを出すだけでアプリは続行（store が nil でも動くようガードされている）。
3. **レガシーデータ移行**：`migrateLegacyData`。旧バージョンが JSON ファイル（`connections.json` / `mcp-settings.json`）で保存していたデータを、DB が空のとき一度だけ取り込む。
4. **接続先読込**：`store.loadConnections()`。DB が存在せず空なら `seedConnections()` で「ローカル確認用」1件を投入。
5. **MCP 設定読込**：`store.loadMCPSettings()`。未保存なら `defaultMCPSettings()`。
6. **環境変数の上書き**：`applyMCPEnv`（待受アドレス・トークン・厳格ホスト鍵）。
7. **サニタイズ**：`sanitizeMCPSettings`（欠損値補完・トークンマスク・`AutoExecuteLowRisk=false` 強制）。
8. **コマンド履歴読込**：`store.loadCommandRequests(500)`。
9. **エージェントポリシー読込**：`store.loadAgentPolicies()` をマップ化。
10. **App 構造体を返す**（各種マップを空初期化）。

この時点ではまだ MCP サーバーは起動していません。

### 5.3 `startup`（`app.go:221`）

Wails がウィンドウ生成後に呼ぶフックです。`ctx` を保持し、`mcp.Enabled` が true なら `startMCPServer()` を呼んで HTTP 待受を開始します。失敗時はステータスを「エラー」にしてログを残します。

### 5.4 `shutdown`（`app.go:239`）

アプリ終了時に呼ばれ、(1) 稼働中のターミナルセッション（SSH/シリアル）を `CloseTerminal()` で閉じ、(2) MCP HTTP サーバーを2秒タイムアウトで graceful shutdown し、(3) SQLite ストアを閉じます。

## 6. バックエンド中核 `app.go` の詳説

`app.go` はおよそ1100行で、本アプリで最も大きく重要なファイルです。役割は「**状態の唯一の所有者**」であり、すべての公開メソッド（Wails 経由で JS から呼ばれる API）と、内部ヘルパーを抱えます。

### 6.1 `App` 構造体（`app.go:24`）

```go
type App struct {
    ctx               context.Context
    mu                sync.RWMutex          // 全フィールドを保護
    store             *Store                // SQLite
    dbPath            string
    legacyConnections string                // 旧 JSON パス（移行用）
    legacyMCPSettings string
    conns             []Connection          // 接続先
    requests          []CommandRequest      // コマンド要求・履歴（新しい順）
    mcp               MCPSettings           // MCP 設定
    mcpToken          string                // Bearer トークン（平文・メモリ）
    mcpServer         *http.Server          // 稼働中の HTTP サーバー
    mcpAgents         map[string]time.Time  // 接続してきたエージェントの最終観測時刻
    agentPolicies     map[string]AgentPolicy// エージェント別の承認省略設定
    sshPassphrases    map[string]string     // 接続先名→パスフレーズ（メモリのみ）
    termMu            sync.Mutex            // ターミナルセッション専用ロック
    term              *terminalSession      // 稼働中のインタラクティブ端末（SSH/シリアル、同時1本）
}
```

> ターミナル機能（第18章）の状態は `term`（現在のセッション）と専用の `termMu` で管理し、本体の `mu` とは独立させています。SSH/シリアルの送受信が長時間続いても、接続先・承認キューなど他の状態アクセスをブロックしないための分離です。アプリ終了時のクリーンアップ（`CloseTerminal`）以外でも、新規セッション確立時に旧セッションを必ず閉じます。

設計上の重要点：

- **単一の `sync.RWMutex` で全状態を守る**。読み取り中心の操作は `RLock`、変更は `Lock`。粒度は粗いが、デスクトップアプリの規模では十分単純で安全。
- **`requests` は新しい順のスライス**。新規要求は先頭に `append([]CommandRequest{req}, a.requests...)` で挿入される（`recordCommandRequest`）。
- **`mcpToken` と `sshPassphrases` は平文でメモリに持つが、SQLite には保存ポリシーが異なる**。トークンは保存される（再起動後も Bearer 認証を維持するため）が、パスフレーズは保存されない。

### 6.2 主要な公開メソッド（Wails API）

これらが `Bind` により JS から呼べる「アプリの操作面」です。

| メソッド | 役割 |
|---|---|
| `GetInitialData()` | 接続先・要求・ポリシー・MCP 設定のスナップショットをまとめて返す。フロントの定期ポーリング元。 |
| `TestConnection(name)` / `TestConnectionWithPassphrase` / `TestConnectionConfig` | SSH 疎通テスト。フォーム上の設定そのままでも、保存済み接続でもテストできる。 |
| `DisconnectConnection(name)` | 接続状態を「Disconnected」に変更（論理的な切断表示）。 |
| `SaveConnection(conn)` | 接続先を追加/更新し SQLite に保存。 |
| `DeleteConnection(name)` | 接続先を削除（履歴は残す）。 |
| `SaveMCPSettings(settings)` | MCP 設定を保存し、待受を再起動。 |
| `DeleteMCPSettings()` | MCP 設定を削除し待受停止。 |
| `GetMCPSettings()` | 現在の MCP 設定スナップショット（接続中エージェント数を再計算）。 |
| `ApproveCommand(id, command)` | 承認キューの要求を実際に SSH 実行する。 |
| `ApproveCommandAndBypassAgent(id, command)` | 承認実行に加え、要求元エージェントへ「承認省略」を付与する。 |
| `RejectCommand(id)` | 要求を「拒否」状態にする。 |
| `DeleteCommandRequest(id)` | コマンド履歴を削除する。 |

### 6.3 接続テストの流れ（`testConnectionConfig`、`app.go:292`）

1. 表示名を決定（名前が空ならホスト名）。
2. パスフレーズの解決：引数で渡されていればそれを使い、なければメモリの `sshPassphrases[conn.Name]` を使う。
3. `ConnectTimeout`（既定10秒）で `context` を生成。
4. `mcp.StrictHostKey` を読み、`testSSHConnection` を呼ぶ。
5. 失敗時は接続ステータスを「Error」にし、`explainSSHError` で人間に分かりやすいエラーメッセージへ翻訳して返す。
6. 成功かつパスフレーズが渡されていた場合は、そのパスフレーズをメモリに記憶（次回以降の実行で使う）。ステータスを「Online」に。

`explainSSHError`（`app.go:756`）は、SSH ライブラリの生エラー文字列を見て「認証失敗」「鍵ファイルが読めない」「パスフレーズ保護」「known_hosts 不一致」などの状況に応じた日本語ガイダンスを付ける関数です。たとえば公開鍵認証に失敗したときは `~/.ssh/authorized_keys` や ssh-agent を確認するよう促し、パスワード認証へ切り替える方法まで案内します。UX に直結する地味だが重要な部分です。

### 6.4 接続先の CRUD

- **`SaveConnection`（`app.go:345`）**：`normalizeConnection` で入力を整形したのち、**種別ごとに必須項目を検証**する。共通で `Name` 必須。`Type=="Serial"` なら `SerialPort` 必須（ホスト/ユーザーは不要）、`Type=="SSH"`（既定）なら `Host`・`User` 必須。検証通過後、メモリ上で同名があれば置換、なければ追加し、SQLite に保存。
- **`DeleteConnection`**：存在確認 → SQLite から削除 → メモリから除去 → `sshPassphrases` も削除。メッセージで「コマンド履歴は監査用に残す」と明示。
- **`normalizeConnection`（`app.go:800`）**：保存前に毎回 `KeyPassphrase=""` にしている点が重要（**パスフレーズは決して接続先レコードに混入しない**）。あわせて `Type` を `connectionType()` で正規化（不正値や空は `SSH` に倒す）、`SerialPort` を trim、シリアルで `BaudRate==0` なら `115200` を補完する。
- **`(Connection) connectionType()`（`app.go:49` 付近）**：`Type` が `"Serial"` ならそのまま、それ以外（空・旧レコード含む）は `"SSH"` を返す小ヘルパー。旧 DB から読んだ種別なしレコードを安全に SSH 扱いにする後方互換の要。

### 6.5 承認と実行（本アプリの心臓部）

#### `ApproveCommand`（`app.go:467`）

人間が GUI で「承認して実行」を押したときに呼ばれます。

1. `id` と `command` の空チェック。GUI 側でコマンド文字列を編集できるため、承認時に渡された `command` を使う（要求時のコマンドと差し替え可能）。
2. `findRequest(id)` で要求を、`findConnection(req.Host)` で接続先を引く。
3. `executeSSH(conn, command)` で実際に SSH 実行。
4. 成功なら状態を「実行済み」、失敗なら「実行失敗」に更新（`updateRequestResult`）。stdout/stderr/所要時間も記録。

#### `ApproveCommandAndBypassAgent`（`app.go:492`）

「このエージェントは今後信頼する」という運用ボタン。要求元エージェント名を取り出し（`未申告エージェント` は不可）、`setAgentApprovalBypass(agentName, true)` でポリシーを立て、続けて `ApproveCommand` を実行します。以後、そのエージェントからの要求は承認キューを経ずに自動実行されるようになります（後述）。

#### `queueCommand` / `recordCommandRequest`（`app.go:609`, `631`）

承認キューへ要求を積む経路。`newCommandRequest` で `REQ-<UnixMilli>-<seq>` 形式の ID、状態「承認待ち」、ユーザー「assistant」、要求時刻（`15:04:05`）などを埋めた `CommandRequest` を作り、メモリ先頭に挿入して SQLite に永続化します。

> **ID の一意化（`seq` サフィックス）**：ID 末尾の `<seq>` はプロセス内で単調増加する `atomic.Uint64`（`app.go` の `requestIDSeq`）です。以前は `REQ-<millis>` のミリ秒精度のみで、承認省略エージェントが並列・連続でコマンドを投げると**同一ミリ秒で ID が衝突**し、SQLite の upsert（`ON CONFLICT(id) DO UPDATE`）が古い行を上書きして履歴が消える不具合がありました。`seq` を足して同一ミリ秒でも必ず一意にすることで解消しています（回帰テスト `TestRapidCommandRequestsPersistDistinctly`）。

#### `executeCommandAndLog`（`app.go:638`）

承認省略エージェント向けの「即時実行＋記録」経路。状態を「自動実行中」で作り、`executeSSH` し、結果に応じて「自動実行済み」/「自動実行失敗」に更新して記録します。

#### `executeSSH`（`app.go:659`）

実行の最終ゲート。`CommandTimeout`（既定30秒）で context を作り、ロック下で `MaxOutputKB`（出力上限）・`StrictHostKey`・該当接続のパスフレーズを読み取り、`runSSHCommand` に委譲します。出力上限が0以下なら `defaultOutputCap`（128KiB）。

### 6.6 エージェントポリシー

- **`setAgentApprovalBypass`（`app.go:574`）**：`AgentPolicy{AgentName, ApprovalBypass, UpdatedAt}` を作り SQLite に保存しメモリ反映。
- **`agentApprovalBypassEnabled`（`app.go:598`）**：MCP ツール処理側がこれを見て「自動実行か承認キューか」を分岐する。

### 6.7 接続中エージェントの計測

`mcpAgents` は「キー（IP＋User-Agent）→最終観測時刻」のマップ。`connectedAgentsLocked`（`app.go:456`）は **直近5分以内** に観測されたエージェントを数えます。これが GUI の「被接続エージェント数」になります。観測は MCP ハンドラの `rememberMCPAgent` で更新されます。

### 6.8 設定の既定値・サニタイズ・環境変数

- **`defaultMCPSettings`（`app.go:940`）**：待受 `127.0.0.1:8787`、パス `/mcp`・`/healthz`・`/audit/events`、Bearer 無効、最大ボディ256KB・最大出力128KB、各タイムアウト、`sudo`/`prod`/書き込み系は承認必須、`AutoExecuteLowRisk=false` など、安全寄りの初期値を返す。
- **`sanitizeMCPSettings`（`app.go:970`）**：空欄に既定値を補完し、`TokenConfigured`/`TokenPreview` を計算、`TokenInput` をクリア、そして **必ず `AutoExecuteLowRisk=false`** を強制。UI からこのフラグを true にする手段はない。
- **`applyMCPEnv`（`app.go:1021`）**：`SSH_GETE_MCP_ADDR`（待受変更）、`SSH_GETE_TOKEN`（トークン設定＋Bearer 有効化）、`SSH_GETE_STRICT_HOSTKEY=1`（厳格ホスト鍵）。
- **`maskToken`（`app.go:1040`）**：トークンを `先頭4...末尾4` 形式でマスク（短い場合は `********`）。GUI には平文を返さない。

### 6.9 レガシー移行とユーティリティ

`migrateLegacyData`（`app.go:834`）は、SQLite が空のとき旧 JSON（`connections.json` / `mcp-settings.json`）を一度だけ取り込みます。`loadConnections`/`saveConnections`/`loadMCPSettings`/`saveMCPSettings` は JSON ファイル入出力のヘルパーで、移行とテスト用に残っています。`cloneConnections`/`cloneRequests`/`cloneAgentPolicies` はロック外へ安全にコピーを渡すためのディープ/シャローコピー関数群です。

## 7. MCP サーバー `mcp_server.go` の詳説

このファイルは、AI エージェントが話す相手である **JSON-RPC 2.0 over HTTP（MCP, Model Context Protocol）** のサーバー実装です。

### 7.1 JSON-RPC 型定義（`mcp_server.go:15`）

`rpcRequest`（jsonrpc/id/method/params）、`rpcResponse`（result または error）、`rpcError`（code/message）という標準的な JSON-RPC 2.0 の型を定義します。`initializeParams`・`toolCallParams`・`executeCommandArgs` がメソッド別のパラメータ型です。

`executeCommandArgs` には `host`/`command`/`reason`/`agent_name`/`requested_by`/`timeout_seconds`/`force_queue` が含まれます（一部は将来用に定義されているが現状未使用のものもある）。

### 7.2 サーバーの起動・再起動（`startMCPServer` / `restartMCPServer`）

`startMCPServer`（`mcp_server.go:53`）は、設定が無効なら「停止中」にして即 return。有効なら：

1. `net.JoinHostPort` で待受アドレスを構築。
2. `http.ServeMux` に3つのハンドラを登録：
   - `HealthPath`（`/healthz`）：`ok/name/mcp/address/connected_agents` を返すヘルスチェック。**認証不要**。
   - `MCPPath`（`/mcp`）：`handleMCP` へ委譲（JSON-RPC 本体）。
   - `AuditPath`（`/audit/events`）：全要求と接続中エージェントを返す監査 API。
3. `http.Server` を各種タイムアウト・最大ヘッダサイズ付きで生成。
4. `net.Listen` 成功後、goroutine で `Serve`。ステータスを「起動中」に。

`restartMCPServer`（`mcp_server.go:113`）は、既存サーバーを2秒で graceful shutdown してから再起動します。設定保存（`SaveMCPSettings`）のたびにこれが呼ばれ、ポートやパスの変更が反映されます。

### 7.3 リクエストハンドリング（`handleMCP`、`mcp_server.go:130`）

1. **CORS プリフライト**：`OPTIONS` には `Access-Control-Allow-Origin: http://localhost` 等を返す。
2. **エージェント記録**：`rememberMCPAgent(r)` で接続元を記録。
3. **メソッド制限**：`POST` 以外は `-32600`。
4. **Bearer 認証**：`BearerEnabled` が true なら `Authorization: Bearer <token>` を検証。不一致は `-32001 unauthorized`。
5. **ボディ上限**：`http.MaxBytesReader` で `MaxBodyKB` を超える本文を拒否。
6. **JSON デコード**：失敗は `-32700 invalid json`。
7. **通知の握りつぶし**：`id` が無く method が `notifications/` で始まるものは `202 Accepted` で即返す（MCP の通知に対応）。
8. **ディスパッチ**：`dispatchMCP(req)` を呼び結果を JSON-RPC レスポンスで返す。

### 7.4 メソッドのディスパッチ（`dispatchMCP`、`mcp_server.go:201`）

- `initialize`：プロトコルバージョン（既定 `2025-06-18`、クライアント指定があれば優先）、`capabilities.tools`、`serverInfo`（name/version）を返す MCP ハンドシェイク。
- `tools/list`：`mcpTools()` が返す4ツールの定義一覧。
- `tools/call`：`callMCPTool` へ。
- それ以外：`-32601 method not found`。

### 7.5 ツールの実行（`callMCPTool`、`mcp_server.go:232`）

4つのツールがあります。

| ツール | 挙動 |
|---|---|
| `list_hosts` | 登録済み接続先の一覧（name/**type**/host/port/user/auth_method/tags/status）を返す。読み取りのみ。`type` でエージェントが SSH/Serial を判別できる。 |
| `get_command_history` | 承認キュー＋履歴を返す。 |
| `request_command_execution` | コマンドを承認キューへ登録。`agent_name` 必須。bypass エージェントなら即実行。 |
| `execute_command` | 同上だが「MCP 経由では Low リスクでも直接実行しない」と明記。bypass エージェントなら即実行。 |

`request_command_execution` と `execute_command` の処理（`mcp_server.go:254` と `301`）はほぼ同じで、流れは：

1. 引数デコード。`host`・`command` 必須。`agent_name`（無ければ `requested_by`）必須。
2. `findConnection(host)` で接続先解決。無ければエラー。
3. **シリアル接続先のガード**：解決した接続先が `Type=="Serial"` なら `serial connections are interactive-only; not executable via MCP` を返して即拒否。シリアルは GUI の対話端末専用で、MCP のコマンド実行モデル（コマンド→終了）には乗らないため。
4. `classifyRisk(command)` でリスク判定。
5. **`agentApprovalBypassEnabled(agentName)` が true なら `executeCommandAndLog` で即実行**し、結果（stdout/stderr/duration）を返す（`auto_executed: true`）。
6. そうでなければ `queueCommand` で承認キューへ積み、`queued: true` と request_id・risk を返す。

つまりエージェント側から見ると、信頼登録されていない限り「キューに積まれた」というレスポンスしか返らず、出力は得られません。人間が承認して初めて実行されます。シリアル接続先は、そもそもこのフローに乗りません（拒否）。

### 7.6 リスク分類（`classifyRisk`、`mcp_server.go:420`）

コマンド文字列を小文字化し、前後に空白を付けた上でパターンマッチします。

- **High**：`rm -`、`mkfs`、`dd if=`、`shutdown`、`reboot`、`systemctl restart/stop`、`iptables`、`userdel`、`chown -r`、`chmod -r` など破壊的・不可逆操作。
- **Medium**：`sudo`、`apt`/`yum`/`dnf`/`brew`、`docker rm/stop`、`kubectl delete`、`mv`、`cp -r`、`truncate` など。
- **Low**：上記以外（読み取り系など）。

これは厳密な構文解析ではなく**ヒューリスティックな文字列マッチ**である点に注意。あくまで人間が承認判断する際の目安として表示するためのもので、これ自体が実行可否を決めるわけではありません（実行可否は承認とポリシーで決まる）。

### 7.7 ツール定義とレスポンス整形

`mcpTools()`（`mcp_server.go:345`）は各ツールの `name`/`description`/`inputSchema`（JSON Schema）を返します。`textToolResult`（`mcp_server.go:398`）は任意の値を整形 JSON にして MCP の `content: [{type:"text", text:...}]` 形式に包みます。`writeJSON`（`mcp_server.go:413`）は `Content-Type` と `MCP-Protocol-Version` ヘッダを付けて JSON を書き出します。

## 8. SSH 実行レイヤ `ssh_exec.go` の詳説

`golang.org/x/crypto/ssh` を使った、純粋に SSH を扱う層です。状態（App）には依存せず、引数で受け取った `Connection` だけで動きます。

### 8.1 出力上限バッファ（`cappedBuffer`、`ssh_exec.go:21`）

巨大な出力でメモリを食い潰さないための仕組み。`max` バイトまで書き込み、超過分は捨てます。`String()` は上限に達していたら末尾に `[ssh-gete: output truncated]` を追記します。stdout/stderr 双方に使われます。

### 8.2 疎通テスト（`testSSHConnection`、`ssh_exec.go:48`）

`openSSHClient` で接続し、すぐ `Close()` するだけ。接続できるかどうかの確認に使います。

### 8.3 コマンド実行（`runSSHCommand`、`ssh_exec.go:56`）

1. クライアント接続 → セッション生成。
2. stdout/stderr を `cappedBuffer` に、stdin を空文字に設定。
3. `session.Run(command)` を goroutine で実行し、結果を channel で待つ。
4. `select` で「context のキャンセル/タイムアウト」と「実行完了」を競争させる。タイムアウト時は `SIGKILL` を送ってセッションを閉じ、`ctx.Err()` を返す。
5. 所要時間・stdout・stderr を `commandResult` に詰めて返す。

context によるタイムアウト制御と、その際の確実な kill がポイントです。

### 8.4 接続確立（`openSSHClient`、`ssh_exec.go:99`）

ポート既定22、接続タイムアウト既定10秒を補完し、認証メソッドとホスト鍵コールバックを用意して `ssh.ClientConfig` を構築。`net.Dialer.DialContext` で TCP を張り（context 対応）、`ssh.NewClientConn` → `ssh.NewClient` で SSH 接続を確立します。

### 8.5 認証メソッド（`sshAuthMethods`、`ssh_exec.go:135`）

認証方式の文字列に「パスワード」が含まれる場合（GUI の「パスワード保管」）は `ssh.Password` を使います。それ以外（鍵認証）は：

1. `agentAuthMethods()` で ssh-agent の鍵を候補に加える。
2. 鍵ファイル（既定 `~/.ssh/id_ed25519`、`expandPath` で `~` 展開）を読む。読めなければ、agent だけで認証を試みる。
3. 鍵をパース。失敗かつパスフレーズがあれば `ParsePrivateKeyWithPassphrase` で再試行。
4. 成功した公開鍵認証を先頭に、agent 認証を後続に並べて返す。

**ssh-agent との併用**（`agentAuthMethods`、`ssh_exec.go:170`）：`SSH_AUTH_SOCK` があれば unix ソケットに接続し、agent の署名鍵を認証候補にします。

### 8.6 ホスト鍵検証（`hostKeyCallback`、`ssh_exec.go:182`）

- 厳格モードでない（かつ `SSH_GETE_STRICT_HOSTKEY != "1"`）場合は `ssh.InsecureIgnoreHostKey()` を返す。**既定はホスト鍵検証なし**。
- 厳格モードなら `~/.ssh/known_hosts` を読み `knownhosts.New` でコールバックを生成。ファイルが無ければエラー。

ローカル開発の利便性を優先した既定値ですが、本番運用では `StrictHostKey` を有効にすべき箇所です（後述の注意点参照）。

### 8.7 パス展開（`expandPath`、`ssh_exec.go:193`）

`~` および `~/...` をユーザーホームに展開する素朴なユーティリティ。

## 9. 永続化レイヤ `storage.go` の詳説

`mattn/go-sqlite3` を使った薄い DAO（データアクセス層）です。`Store` 構造体が `*sql.DB` とパスを持ちます。

### 9.1 オープンとマイグレーション（`openStore` / `migrate`）

`openStore`（`storage.go:21`）は、DB ファイルの親ディレクトリを `0700` で作成し、`?_busy_timeout=5000&_foreign_keys=on` 付きで開いて `migrate()` を呼びます。

`migrate`（`storage.go:44`）は `CREATE TABLE IF NOT EXISTS` を並べた素朴な方式（バージョン管理されたマイグレーションではない）。`PRAGMA journal_mode=WAL` で並行性を確保。テーブルは4つ：

- **`connections`**：接続先（name が主キー、tags は JSON 文字列で保存）。SSH/シリアル両対応のため `type`・`serial_port`・`baud_rate` カラムを持つ。
- **`mcp_settings`**：単一行（`id=1` 固定）に設定 JSON とトークンを格納。
- **`command_requests`**：コマンド要求・履歴。`created_at`（UnixMilli の INTEGER）で並べ替え。
- **`agent_policies`**：エージェント別の承認省略フラグ。

**追加カラムのマイグレーション**：`type`・`serial_port`・`baud_rate` は後から増えたカラムなので、`CREATE TABLE` 直後に `addColumnIfMissing`（`storage.go`）で「無ければ `ALTER TABLE ADD COLUMN`」を行う。`pragma_table_info` で既存カラムを調べ、存在しないものだけ追加するため、旧 DB も新 DB も同じコードで安全に移行できる（`type` は既定 `'SSH'`、`baud_rate` は既定 `0`）。バージョン管理されたマイグレーションへの第一歩。

### 9.2 各テーブルの CRUD

いずれも `INSERT ... ON CONFLICT(...) DO UPDATE`（upsert）パターンを使い、同一キーなら更新します。

- **接続先**：`loadConnections`（name 昇順）/`saveConnection`（tags を JSON 化）/`deleteConnection`。
- **MCP 設定**：`loadMCPSettings`（行が無ければ既定値）/`saveMCPSettings`（保存前に `TokenInput`/`TokenPreview` をクリア）/`deleteMCPSettings`。
- **コマンド要求**：`loadCommandRequests(limit)`（既定500件、`created_at DESC`）/`saveCommandRequest`（ID から作成時刻 millis を復元し `created_at` に使う）/`deleteCommandRequest`。
- **エージェントポリシー**：`loadAgentPolicies`（bool は 0/1 で保存）/`saveAgentPolicy`。

`parseRequestIDMillis`（`storage.go` 末尾）は `REQ-` プレフィックスを外し、**先頭のミリ秒部分のみ**を数値化して要求の作成時刻を ID から導出します。ID が `REQ-<millis>-<seq>`（新形式）でも `REQ-<millis>`（旧形式）でも、最初の `-` までを見ることで両対応しています。これにより並べ替え用のタイムスタンプを別途持たずに済みます。

### 9.3 永続化ポリシーのまとめ

- 接続先の `credential`（鍵パスやパスワード）は**保存される**。
- MCP の Bearer トークンも**保存される**（再起動後も認証維持のため）。
- SSH 鍵の**パスフレーズは保存されない**（メモリのみ）。
- コマンド履歴は接続先を消しても**残る**（監査目的）。

## 10. フロントエンド `frontend/src/main.js` の詳説

約1465行の単一ファイルで、フレームワークなしの SPA を実装しています。構造は「状態 → 描画関数 → イベントハンドラ」の3部構成です（加えて xterm.js のターミナル統合を持つ）。

### 10.1 起動と import（`main.js:1`）

AdminLTE の CSS/JS、Bootstrap Icons、自前 `style.css` を import し、Wails 生成バインディングから Go メソッド（`GetInitialData`、`SaveConnection` など）を import します。`document.body.className` でレイアウトクラスを設定し、`#app` 要素を取得します。

### 10.2 グローバル状態（`state`、`main.js:24`）

単一のプレーンオブジェクトに全 UI 状態を持ちます。

- `active`：現在のタブ（`connections`/`commands`/`terminal`/`settings`）。
- `selectedConnection` / `selectedRequest`：選択インデックス。
- `data`：バックエンドから取得した接続先・要求・ポリシー・MCP 設定。
- `dirty`：フォーム編集中フラグ（ポーリングで上書きしないため）。
- `passphrase` / `deleteConnection` / `deleteCommand`：各モーダルの開閉と入力値。
- `connectionTest`：直近の疎通テスト結果。
- `terminal`：ターミナル接続状態（`connected` / `label`）。
- `serialPorts`：接続先編集（シリアル）で選ばせる検出済みシリアルポート一覧。
- `toast`：トースト通知文字列。

加えて、`state` の外側にモジュール変数として **xterm インスタンス（`term`）・FitAddon（`fitAddon`）・配線済みフラグ（`termWired`）** を保持します。これらは全画面再描画（`innerHTML` 全置換）で破棄されないよう、あえて `state`/DOM の外に置いています（第18章で詳述）。

### 10.3 描画モデル

レンダリングは「**状態から HTML 文字列を組み立て、`app.innerHTML` に丸ごと代入する**」方式。`render()` が `active` に応じて `renderConnections`/`renderCommands`/`renderTerminal`/`renderSettings` を呼び、`shell(content)` でサイドバー・ヘッダー・モーダルごと全体を生成します。`active === 'terminal'` のときは描画後に `mountTerminal()` を呼び、保持してある xterm の DOM 要素を新しいホストへ再アタッチします（第18章）。

仮想 DOM は無いため、毎回フル再構築されます。`escapeHtml`（`main.js:66`）で値を必ずエスケープし、XSS を防いでいます。すべての描画関数（`connectionRow`、`commandDetail`、`historyTable`、`inputCol`、`selectCol`、`checkCol`、`terminalPanel` 等）はテンプレートリテラルを返す純関数です。

### 10.4 4つの画面

- **接続先管理（`renderConnections`）**：左に接続先一覧（SSH/Serial バッジ付き）、右に編集フォーム。先頭に **接続種類セレクタ（SSH / Serial）** があり、種別に応じてフォームが切り替わる。SSH ならホスト・ポート・ユーザー・認証方式・資格情報・タイムアウトを、Serial ならシリアルポート（検出ドロップダウン＋再スキャン）・ボーレートを表示。共通で名前・タグ・説明。「保存」ボタン（SSH のみ「接続」「切断」の疎通テストボタンも）。
- **コマンド履歴・承認（`renderCommands`）**：左に承認キュー（`status === '承認待ち'` のみ）、右に選択中要求の詳細（エージェント名・コマンド・理由・stdout/stderr）。下部に全履歴テーブル。「拒否」「エージェントは承認省略」「承認して実行」ボタン。
- **ターミナル（`renderTerminal`）**：左に接続先選択（種別バッジ付き）＋詳細＋「接続/切断」、右に xterm.js のライブ端末。選択した接続先の **種別に従って** SSH 対話シェル or シリアルコンソールを開く（トグルではない）。第18章で詳述。
- **MCP 待受設定（`renderSettings`）**：左に全設定フォーム（待受・認証・パス・タイムアウト・承認ポリシー等）、右に公開エンドポイント情報とエージェント用設定 JSON（`agentConfig`）・ツール usage 一覧。

`agentConfig`（`main.js:115`）は、エージェント側の `mcpServers` 設定 JSON を生成します。Bearer 有効時は `Authorization: Bearer ${SSH_GETE_TOKEN}` ヘッダを含めます。これをコピーしてエージェントに貼るだけで接続設定が済みます。

### 10.5 イベント処理（`main.js:868` 以降）

`#app` に対する委譲（event delegation）で、クリック・サブミット・入力・変更を一括処理します。

- **クリック**：`data-nav`（タブ切替）と `data-action`（各操作）を見て分岐。`new-connection`、`connect-ssh`、`disconnect-ssh`、各種削除モーダルの開閉・確定、コピー系（`copy-command`/`copy-mcp-endpoint`/`copy-agent-config`）、`approve-command`、`approve-bypass-agent`、`reject-command`、`refresh-mcp`、`delete-mcp` などを処理。
- **サブミット**：`connection-form`（保存）、`mcp-form`（MCP 設定保存）、`passphrase-form`（パスフレーズ付き疎通テスト）。`event.preventDefault()` でページ遷移を防ぐ。
- **入力/変更**：フォームに触れたら `dirty` フラグを立て、ポーリングによる上書きを防ぐ。

### 10.6 パスフレーズの扱い（UX）

「接続」押下時、バックエンドが `passphrase protected` を含むエラーを返したら、`passphrase` モーダルを開いてパスフレーズを尋ねます（`main.js:907`）。入力後 `TestConnectionConfig(connection, passphrase)` で再試行します。パスフレーズはモーダル内でのみ扱われ、成功時にバックエンドのメモリに記憶されます。

### 10.7 データ同期（`refreshData`、`main.js:152`）

`GetInitialData()` を呼んで `state.data` を更新します。重要なのは **編集保護**：

- 接続先タブで編集中（`dirty.connections`）なら接続先は上書きしない。
- 設定タブで編集中（`dirty.settings`）なら MCP 設定は上書きしない。
- いずれかのモーダルが開いていれば再描画しない。
- **ターミナルタブ表示中（`protectTerminal`）は再描画しない**（`state.data` は更新するが `render()` は呼ばない）。これがないと5秒ごとに xterm が作り直され、対話セッションが壊れる。

これにより、5秒ごとの自動ポーリング（末尾の `setInterval`）がユーザーの入力やライブ端末を踏み潰さないようになっています。要求（承認キュー）は編集対象でないため常に最新化されます。

### 10.8 補助関数

`copyText`（`main.js:123`）はクリップボード API を使い、失敗時は隠し textarea にフォールバック。`showToast`（`main.js:142`）は3.2秒で自動的に消えるトーストを出します。ステータスバッジ生成（`mcpStatusBadge`/`sshStatusBadge`）やフォーム値収集（`collectConnectionForm`/`collectMCPForm`）も用意されています。`collectMCPForm` では `autoExecuteLowRisk: false` が常にハードコードされており、バックエンドのサニタイズと二重に安全側へ倒しています。

## 12. データモデル一覧

`app.go` で定義される主要構造体（JSON タグ＝フロント/MCP との連携キー）。

```
Name, Type(SSH/Serial), Host, Port, User, AuthMethod, Credential,

Tags[], Description, Status, LastChecked
```

`Type` で SSH/シリアルを区別。SSH なら Host/User/AuthMethod/Credential 系、シリアルなら SerialPort/BaudRate を使う。`connectionType()` ヘルパーが空・不正値を `SSH` に正規化（後方互換）。

```
ID(REQ-<millis>-<seq>), Host, User, Command, Reason, RequestedBy,

Status(承認待ち/実行済み/実行失敗/拒否/自動実行済み/自動実行失敗),

```
ID は `REQ-<ミリ秒>-<プロセス内連番>`。連番で同一ミリ秒の衝突を防ぐ。

### `terminalSession`（インタラクティブ端末・非永続）
```
kind(ssh/serial), label, stdin(io.Writer),
resize(func(cols,rows)error・任意), closer(func())
```
SSH/シリアルを同一インターフェイスで扱うための内部型（`terminal.go`）。永続化はされず、稼働中のみ `App.term` に1本だけ保持。

### `AgentPolicy`（エージェント別ポリシー）
```
AgentName, ApprovalBypass(bool), UpdatedAt
```

### `MCPSettings`（MCP 待受設定）
待受（Enabled/ListenAddress/Port/Transport/BaseURL）、パス（MCPPath/HealthPath/AuditPath）、認証（BearerEnabled/TokenName/TokenInput/TokenPreview/TokenConfigured）、ネットワーク（AllowedOrigins/TLSMode/ProxyMode）、上限（MaxBodyKB/MaxOutputKB/RequestTimeout）、既定タイムアウト、セキュリティ（StrictHostKey/RequireApprovalSudo/Prod/WriteOp、AutoExecuteLowRisk）、状態（Status/LastError/ConnectedAgents）。

`GetInitialData()` の戻り値。Connections/Requests/AgentPolicies/MCP をまとめたもの。

### `ActionResult`
`OK(bool)` と `Message(string)`。ほぼ全ての公開メソッドが返す共通結果型で、フロントはこれを使ってトースト表示する。

## 14. セキュリティ設計と承認モデル

本アプリの価値の中心はセキュリティ設計です。要点を整理します。

### 14.1 多層の安全装置

1. **デフォルト承認必須**：MCP 経由のコマンドはリスクに関係なく承認キュー行き。`AutoExecuteLowRisk` はバックエンド（`sanitizeMCPSettings`）とフロント（`collectMCPForm`）の両方で false に固定され、UI から有効化できない。
2. **エージェント名の必須化**：`agent_name`（または `requested_by`）が無い要求は拒否。誰の要求かを必ず記録する。
3. **明示的な信頼付与**：承認省略は、人間が GUI で特定エージェントに対し意図的に付与したときだけ有効。`未申告エージェント` には付与不可。
4. **リスク可視化**：`classifyRisk` が High/Medium/Low を付け、人間の判断を助ける。
5. **出力上限**：`cappedBuffer` で stdout/stderr を `MaxOutputKB` に制限。
6. **タイムアウトと kill**：接続・コマンドそれぞれにタイムアウトがあり、超過時は SIGKILL。
7. **監査ログ**：全要求が SQLite に残り、接続先を消しても履歴は保持。`/audit/events` で外部からも取得可能。

### 14.2 ネットワーク面

- 既定待受は `127.0.0.1`（ローカルのみ）。
- Bearer 認証（任意）。トークンは GUI にマスク表示され、平文は返さない。
- CORS は `http://localhost` と `wails://wails.localhost` を想定。
- ボディサイズ上限・各種 HTTP タイムアウトを設定。

### 14.3 資格情報の保持方針

- SSH 鍵パスフレーズはメモリのみ（DB 非保存）。
- Bearer トークンは DB 保存（再起動後の認証維持）。
- 接続先の credential（鍵パス/パスワード）は DB 保存。

## 16. 既知の注意点・改善余地

コードを読む際に意識しておくとよい点を挙げます（批判ではなく、改修時のヒント）。

1. **ホスト鍵検証が既定オフ**：`hostKeyCallback` は厳格モードでなければ `InsecureIgnoreHostKey()` を返す。中間者攻撃を考えると本番では `StrictHostKey` を有効にしたい。ローカル前提のアプリゆえの割り切り。
2. **リスク分類はヒューリスティック**：`classifyRisk` は文字列マッチで、`rm`をクオートで隠す、エイリアスを使う等で回避され得る。あくまで人間の判断補助であり、これに依存して自動実行可否を決める設計にはなっていない点が安全。
3. **承認省略の強さ**：bypass を付与したエージェントは以後ノーチェックで実行できる。付与は慎重に。
4. **DB マイグレーションは簡易方式**：基本は `CREATE TABLE IF NOT EXISTS`。シリアル対応で増えた `connections` のカラム（`type`/`serial_port`/`baud_rate`）は `addColumnIfMissing`（`pragma_table_info` を見て無ければ `ALTER TABLE ADD COLUMN`）で追加しており、旧 DB も移行できる。ただしバージョン番号付きの本格的なマイグレーション管理ではないため、複雑なスキーマ変更時は別途設計が要る。
5. **トークンは平文で DB 保存**：ローカルの単一ユーザー前提。DB ファイル自体の保護（`0700` ディレクトリ）に依存。
6. **`force_queue` / `timeout_seconds` など未使用フィールド**：`executeCommandArgs` に将来用の項目がある。
7. **粗粒度ロック**：単一 `RWMutex` で全状態を保護。規模的に問題ないが、長時間の SSH 実行を排他区間に含めない設計（`executeSSH` 内ではロックを早めに解放している）になっている点は良い。

## 17. 用語集

- **MCP（Model Context Protocol）**：AI エージェントが外部ツール/データに接続するためのプロトコル。本アプリは JSON-RPC 2.0 over HTTP（Streamable HTTP）で実装。
- **Wails**：Go バックエンドと Web フロントを単一デスクトップアプリに統合するフレームワーク。Go メソッドを JS から直接呼べる。
- **承認キュー**：エージェントが要求したコマンドのうち、人間の承認待ちの状態にあるものの一覧。
- **承認省略（bypass）**：特定エージェントに対し、承認なしの即時実行を許可するポリシー。
- **リスク分類**：コマンド文字列から High/Medium/Low を推定する仕組み。
- **シード（seed）**：DB が空のとき投入される初期データ（「ローカル確認用」接続）。
- **サニタイズ**：MCP 設定の欠損補完・安全側固定（`AutoExecuteLowRisk=false` 等）を行う処理。
- **PTY（疑似端末）**：対話シェルに必要な仮想端末。`vim`/`top` やタブ補完が動く前提。SSH では `RequestPty` で要求する。
- **トランスポート抽象化**：SSH とシリアルを `terminalSession` という共通インターフェイスで扱い、同じ画面・同じ操作で動かす設計。
- **xterm.js**：ブラウザ上のターミナルエミュレータ。バックエンドのバイト列をそのまま描画し、キー入力を送り返す。

## 18. インタラクティブターミナルとシリアル接続（`terminal.go`）

「コマンドを承認キューに積む」MCP 経路とは別に、本アプリは **人間が GUI で直接操作する対話ターミナル** を持ちます。SSH の PTY シェルと、USB シリアル等のシリアルポートの両方に、同じ画面（xterm.js）・同じ操作で接続できます。実装の中心は新規ファイル `terminal.go`（270行）と、`frontend/src/main.js` の xterm 統合です。

> 位置づけの整理：**MCP 経路はエージェント用で承認フローに乗る**、**ターミナル経路は人間用で対話的**、という役割分担です。ターミナルは GUI 操作者＝承認者という前提で、承認キューを介さず直接つなぎます。そのためシリアル接続先は MCP からは実行できません（第7章のガード）。

### 18.1 設計の核：トランスポート非依存の `terminalSession`

```go
type terminalSession struct {
    kind   string                       // "ssh" | "serial"
    label  string                       // 表示用ラベル
    stdin  io.Writer                    // ユーザー入力の書き込み先
    resize func(cols, rows int) error   // 端末サイズ変更（シリアルは nil）
    closer func()                       // 後始末
}
```

SSH も シリアルも、結局は「**入力を書き込む先（`stdin`）**」「**出力を読み出してフロントへ流す**」「**閉じる（`closer`）**」の3点に抽象化できます。`terminalSession` はその最小インターフェイスで、SSH/シリアルそれぞれの起動関数がこの構造体を1つ組み立てます。これにより `SendTerminalInput` / `ResizeTerminal` / `CloseTerminal` は**トランスポートを知らずに**動きます。

稼働中のセッションは `App.term` に**1本だけ**保持し、`termMu` で保護します。`setTerminalSession`（新セッション差し込み時に旧セッションを必ず閉じる）が排他の要です。

### 18.2 出力ストリーミング：Wails イベント

リモート出力はポーリングではなく **Wails のイベント**でフロントへ push します。

```go
type ptyEmitter struct{ app *App }

func (e *ptyEmitter) Write(p []byte) (int, error) {
    if e.app != nil && e.app.ctx != nil {
        wruntime.EventsEmit(e.app.ctx, terminalDataEvent /* "terminal:data" */,
            base64.StdEncoding.EncodeToString(p))
    }
    return len(p), nil
}
```

- `ptyEmitter` は `io.Writer`。SSH は `session.Stdout`/`session.Stderr` にこれを差し込み、シリアルは読み取りループから `Write` を呼ぶ。どちらも同じ経路でフロントへ届く。
- **base64 エンコード**がポイント。端末出力は任意のバイト列（不正な UTF-8 や制御コードを含む）で、JSON イベントペイロードにそのまま載せると壊れる。base64 にすればバイナリ安全に運べる。フロントは `atob` で復号して `term.write` する。
- セッション終了時は `terminal:exit` イベントを送り、フロントが「切断」表示に切り替える。

### 18.3 SSH 側：`startSSHSession`

```go
func (a *App) StartTerminal(name string) ActionResult {
    conn, ok := a.findConnection(name)
    if !ok { return ActionResult{OK: false, Message: "対象の接続先が見つかりません"} }
    if conn.connectionType() == "Serial" { return a.startSerialSession(conn) }
    return a.startSSHSession(conn)
}
```

`StartTerminal` は**接続先の種別でディスパッチ**するだけ。SSH 側の `startSSHSession` は：

1. `RLock` で `StrictHostKey` とメモリ上のパスフレーズを取り出す（既存の `executeSSH` と同じ作法）。
2. `openSSHClient`（`ssh_exec.go` を再利用）で接続。**ダイヤル用 context は接続確立だけを制限**し、その後 `cancel()` してもクライアントは生き続ける。
3. `session.RequestPty("xterm-256color", 24, 80, modes)` で PTY を確保（`modes` は `ECHO` 等）。これにより `vim`/`top`/タブ補完が動く本物のシェルになる。
4. `StdinPipe()` を `terminalSession.stdin` に、`ptyEmitter` を stdout/stderr に設定し `session.Shell()` 起動。
5. `resize` は `session.WindowChange(rows, cols)` を呼ぶクロージャ。
6. goroutine で `session.Wait()` を待ち、終了したら `App.term` を片付けて `terminal:exit` を送る。

### 18.4 シリアル側：`ListSerialPorts` と `startSerialSession`

```go
func (a *App) ListSerialPorts() []string {
    ports, err := serial.GetPortsList()
    if err != nil || ports == nil { return []string{} }
    return ports
}
```

- `ListSerialPorts` は `go.bug.st/serial` でポートを列挙。接続先編集のシリアルポート選択ドロップダウンに使う（macOS は `/dev/cu.usbserial-*` 等）。
- `startSerialSession` は `serial.Open(portName, &serial.Mode{BaudRate: baud})` でポートを開き、`terminalSession` を組む。**`stdin` はシリアルポートそのもの**（`io.ReadWriteCloser`）、`resize` は `nil`（シリアルに端末サイズの概念はない）、`closer` は `port.Close()`。
- 出力は専用 goroutine の**読み取りループ**：`port.Read(buf)` で読んだ分を `ptyEmitter.Write` でフロントへ。`Read` がエラーを返したらループを抜け、`App.term` を片付けて `terminal:exit` を送る。
- ポート/ボーレートは接続先（`conn.SerialPort` / `conn.BaudRate`）から取る。空なら旧データ互換で `Host` をデバイスパスとして許容し、ボーレート未指定は 115200 にフォールバック。

### 18.5 入力・リサイズ・切断（トランスポート共通）

```go

func (a *App) ResizeTerminal(cols, rows int) ActionResult { /* ts.resize があれば呼ぶ。無ければ no-op */ }
func (a *App) CloseTerminal() ActionResult               { /* a.term を nil にして closeQuiet() */ }
```

- キー入力（xterm の `onData`）は UTF-8 文字列としてそのまま `stdin` に書く。SSH でもシリアルでも同じ。
- `ResizeTerminal` はシリアルでは `resize == nil` なので安全に no-op。SSH のみ実効。
- `CloseTerminal` はアプリ終了時（`shutdown`）にも呼ばれる。

### 18.6 フロント側の難所：全画面再描画の中で xterm を生かす

本アプリのフロントは「状態が変わるたび `app.innerHTML` を丸ごと差し替える」方式です（第10章）。素直に書くと、5秒ポーリングや他操作の再描画のたびに xterm の DOM が破棄され、対話セッションが壊れます。これを次の2段構えで防いでいます。

1. **xterm インスタンスを `state`/DOM の外（モジュール変数 `term`）に保持**し、`render()` 後の `mountTerminal()` で「既存の `term.element` を新しいホスト要素へ再アタッチ（`appendChild`）」する。再生成ではなく**移植**なので、スクロールバックも生きたまま。
2. **ターミナルタブ表示中はポーリングによる `render()` を抑止**（`refreshData` の `protectTerminal`）。そもそも余計な再描画を起こさない。

出力は `EventsOn('terminal:data', payload => term.write(b64ToBytes(payload)))`、終了は `EventsOn('terminal:exit', …)` で受けます。`window` の `resize` で `fitAddon.fit()` ＋ `ResizeTerminal` を呼び、サイズを追従させます。タブを離れる/切断時は `teardownTerminal()` で `CloseTerminal()` ＋ `term.dispose()`。

### 18.7 接続先に種別を持たせる（UI の自然さ）

ターミナルの SSH/シリアル切替は**トグルではなく、接続先の属性**で決まります。接続先編集に「接続種類（SSH / Serial）」セレクタがあり、選んだ種別でフォームが切り替わります（`connectionTypeCol` / `connectionSSHFields` / `connectionSerialFields`）。ターミナル画面は選択中の接続先の `type` に**従属**し、`StartTerminal(name)` がバックエンドで自動ディスパッチします。これにより「この機器はシリアル」という事実を一度登録すれば、以後は接続先を選ぶだけで正しい方式で開けます。

### 18.8 データフロー：シリアルコンソールを開く

```
1. 接続先管理で 種別=Serial・ポート=/dev/cu.usbserial-XXXX・ボーレートを保存
2. ターミナル画面でその接続先を選び「シリアル接続」→ StartTerminal(name)
3. connectionType()=="Serial" → startSerialSession → serial.Open
4. 読み取り goroutine: port.Read → ptyEmitter(base64) → "terminal:data"
5. フロント: EventsOn('terminal:data') → term.write（画面に出力）
6. キー入力 → term.onData → SendTerminalInput → port.Write（機器へ）
7. 「切断」→ teardownTerminal → CloseTerminal → port.Close → "terminal:exit"
```

SSH 対話シェルも、3〜6 が `startSSHSession`/`session.Shell()`/`StdinPipe` に変わるだけで、フロントの受け口は完全に同じです。これがトランスポート抽象化の利き目です。

### A-2. `executeSSH()`（app.go:659〜671）— ロック設計のお手本

```go
func (a *App) executeSSH(conn Connection, command string) (commandResult, error) {
    ctx, cancel := context.WithTimeout(context.Background(), timeoutSeconds(conn.CommandTimeout, 30))
    defer cancel()
    a.mu.RLock()
    outputCap := a.mcp.MaxOutputKB * 1024
    strictHostKey := a.mcp.StrictHostKey
    conn.KeyPassphrase = a.sshPassphrases[conn.Name]
    a.mu.RUnlock()
    if outputCap <= 0 { outputCap = defaultOutputCap }
    return runSSHCommand(ctx, conn, command, outputCap, strictHostKey)
}
```

この短い関数には設計の良さが凝縮されています。

1. **ロック区間が極小**：`RLock`〜`RUnlock` の中では「設定値の読み取り」と「パスフレーズの転記」だけを行い、**時間のかかる `runSSHCommand`（実際の SSH 通信）はロックの外**で実行する。これにより、長時間コマンドが他のすべての状態アクセスをブロックしない。
2. **`conn` は値渡し**なので、`conn.KeyPassphrase` に書き込んでも共有状態を汚さない。ローカルコピーにパスフレーズを差し込んでから渡している。
3. `timeoutSeconds(conn.CommandTimeout, 30)` で 0/負値を 30 秒にフォールバック。
4. `outputCap` が 0 以下なら 128KiB に補正。

### C-1. `cappedBuffer`（ssh_exec.go:21〜46）

```go
func (b *cappedBuffer) Write(p []byte) (int, error) {
    if b.max <= 0 { return len(p), nil }            // 上限0なら全捨て（書いたことにする）
    remaining := b.max - b.buf.Len()
    if remaining > 0 {
        if len(p) > remaining { _, _ = b.buf.Write(p[:remaining]) } else { _, _ = b.buf.Write(p) }
    }
    return len(p), nil                               // 常に len(p) を返す＝書き手をブロックしない
}
```

- `Write` は **常に `len(p)` を返す**。これにより、上限到達後も SSH セッションの出力読み取りが「全部書けた」と認識し、エラーや停止を起こさずコマンドが最後まで走る。捨てるのは内部バッファへの蓄積だけ。
- `String()` は上限到達時に `[ssh-gete: output truncated]` を付け、切り詰めが起きたことを人間に知らせる。

### C-2. `runSSHCommand()` の select 競争（ssh_exec.go:76〜96）

```go
    done := make(chan error, 1)          // バッファ付き＝goroutine が送信でブロックしない
    go func() { done <- session.Run(command) }()

    select {
    case <-ctx.Done():
        _ = session.Signal(ssh.SIGKILL)
        _ = session.Close()
        return commandResult{Stdout: stdout.String(), Stderr: stderr.String(), Duration: time.Since(start)}, ctx.Err()
    case err := <-done:
        return commandResult{Stdout: stdout.String(), Stderr: stderr.String(), Duration: time.Since(start)}, err
    }
```

- `done` をバッファ 1 にしているのは、タイムアウト側（`ctx.Done()`）が勝った場合でも、後から `session.Run` が完了して `done` に送信する際に goroutine がリークしないようにするため。
- タイムアウト時は `SIGKILL` を送ってからセッションを閉じ、`ctx.Err()`（`context.DeadlineExceeded` など）を返す。途中までの出力も返す点が親切。

### C-3. `sshAuthMethods()` の優先順位（ssh_exec.go:135〜168）

認証方式の組み立て順は「**公開鍵を先頭、ssh-agent を後続**」です。

```go
    return append([]ssh.AuthMethod{ssh.PublicKeys(signer)}, auth...), nil
```

- SSH サーバーは提示された認証メソッドを順に試すため、明示指定した鍵を最優先にし、ダメなら agent の鍵群にフォールバックする設計。
- 鍵ファイルが読めない場合でも、agent が使えるならそれだけで認証を試みる（`if len(auth) > 0 { return auth, nil }`）。
- パスフレーズ保護鍵は、まず無パスフレーズでパースを試み、失敗かつ `KeyPassphrase` があれば `ParsePrivateKeyWithPassphrase` で再挑戦。

### C-4. `hostKeyCallback()` の既定挙動（ssh_exec.go:182〜191）

```go
    if !strict && os.Getenv("SSH_GETE_STRICT_HOSTKEY") != "1" {
        return ssh.InsecureIgnoreHostKey(), nil
    }
```

- 設定とフラグ環境変数の **両方** が無効のときだけ検証をスキップ。どちらか一方でも厳格を要求すれば `known_hosts` 検証に切り替わる。
- ローカル利便性のための既定オフだが、本番ではオンにすべき箇所（付録の注意点で再掲）。

## 付録D：`storage.go` 逐行コードリーディング

### D-1. `openStore()` の DSN（storage.go:21〜35）

```go
    db, err := sql.Open("sqlite3", path+"?_busy_timeout=5000&_foreign_keys=on")
```

- `_busy_timeout=5000`：別接続がロック中でも 5 秒待ってからエラーにする（WAL と併せて並行性を確保）。
- `_foreign_keys=on`：外部キー制約を有効化（将来のテーブル間制約に備える）。
- ディレクトリは `0o700` で作成し、所有者のみアクセス可に。

### D-2. upsert パターン（storage.go 全体に共通）

すべての save 系は `INSERT ... ON CONFLICT(key) DO UPDATE SET col = excluded.col` 形式です。

```go
    INSERT INTO connections (...) VALUES (...)
    ON CONFLICT(name) DO UPDATE SET host = excluded.host, port = excluded.port, ...
```

- `excluded.X` は「INSERT しようとした新しい値」を指す SQLite の構文。
- これにより「新規なら挿入、既存なら更新」を 1 文で表現でき、アプリ側で存在チェックを分岐しなくてよい。

### D-3. ID からのタイムスタンプ復元（storage.go:222〜249, 295〜297）

```go
func (s *Store) saveCommandRequest(req CommandRequest) error {
    createdAt := time.Now().UnixMilli()
    if parsed, err := parseRequestIDMillis(req.ID); err == nil { createdAt = parsed }
    ...
}

// IDs are "REQ-<millis>-<seq>"; legacy IDs are "REQ-<millis>" (no seq).
func parseRequestIDMillis(id string) (int64, error) {
    rest := strings.TrimPrefix(id, "REQ-")
    if idx := strings.IndexByte(rest, '-'); idx >= 0 { rest = rest[:idx] }
    return strconv.ParseInt(rest, 10, 64)
}
```

- 要求 ID は `REQ-<UnixMilli>-<seq>` 形式（`<seq>` はプロセス内連番。同一ミリ秒での ID 衝突＝履歴消失を防ぐ）。`parseRequestIDMillis` は**最初の `-` までのミリ秒部分だけ**を取り出すので、新形式でも旧 `REQ-<millis>` でも作成時刻を復元できる。
- 復元した時刻を `created_at` 列に入れ、`loadCommandRequests` の `ORDER BY created_at DESC` で**生成順（新しい順）**に並べる。
- ID をパースできない異常系では「今の時刻」にフォールバック。

### D-4. `loadMCPSettings()` の「行なし＝既定」処理（storage.go:155〜171）

```go
    err := s.db.QueryRow(`SELECT payload_json, token FROM mcp_settings WHERE id = 1`).Scan(&payload, &token)
    if errors.Is(err, sql.ErrNoRows) { return settings, "", false, nil }   // 既定＋loaded=false
```

- 戻り値の3つ目 `loaded bool` で「DB に設定が存在したか」を呼び出し側に伝える。`NewApp` はこれを見て「未保存なら既定値」を確定する。

## 付録E：`frontend/src/main.js` 逐行コードリーディング

### E-1. 全画面再描画モデル（main.js:859〜866）

```js
function render() {
  const content = state.active === 'connections'
    ? renderConnections()
    : state.active === 'commands'
      ? renderCommands()
      : renderSettings();
  app.innerHTML = shell(content);
}
```

- 仮想 DOM なし。状態が変わるたびに HTML 文字列を作り直して `innerHTML` に丸ごと代入する。
- 副作用として、再描画のたびに DOM が作り直されるため、フォーカスや未送信入力が失われ得る。これを補うのが後述の `dirty` フラグと「編集中はポーリング再描画を抑止」する仕組み。

### E-2. XSS 対策（main.js:66〜73）

```js
function escapeHtml(value) {
  return String(value ?? '')
    .replaceAll('&', '&amp;').replaceAll('<', '&lt;').replaceAll('>', '&gt;')
    .replaceAll('"', '&quot;').replaceAll("'", '&#039;');
}
```

- テンプレートリテラルに値を埋め込む箇所はほぼ例外なく `escapeHtml` を通す。コマンド文字列・stdout・エージェント名など、外部（エージェント）由来の文字列を画面に出すため、エスケープは必須。`&` を最初に置換するのは二重エスケープ防止の定石。

### E-3. 編集保護つきポーリング（main.js:152〜172, 1135）

```js
async function refreshData({ silent = false } = {}) {
  const next = await GetInitialData();
  const protectConnections = state.active === 'connections' && state.dirty.connections;
  const protectSettings = state.active === 'settings' && state.dirty.settings;
  const protectModal = state.passphrase.open || state.deleteConnection.open || state.deleteCommand.open;
  state.data = {
    connections: protectConnections ? state.data.connections : next.connections,
    requests: next.requests,                          // 要求は常に最新化
    agentPolicies: next.agentPolicies ?? [],
    mcp: protectSettings ? state.data.mcp : next.mcp,
  };
  ...
  if (!protectConnections && !protectSettings && !protectModal) render();
}
...
window.setInterval(() => refreshData({ silent: true }), 5000);
```

- 5 秒ごとに自動同期するが、**ユーザーがフォーム編集中（dirty）やモーダル表示中は、その領域を上書きせず再描画もしない**。これで「入力中に画面がリセットされる」事故を防ぐ。
- 承認キュー（`requests`）は編集対象でないため常に最新へ。新着の承認待ちがすぐ反映される。
- `silent` は失敗時にエラー画面を出すかどうかの制御（初回 `refreshData()` は非 silent で、致命的失敗を画面表示する）。

### E-4. パスフレーズのフロー（main.js:904〜925, 1105〜1117）


- まずパスフレーズなしで疎通テストし、バックエンドが `passphrase protected` を含むエラーを返したときだけモーダルを開く。

- パスフレーズはフロントの `state.passphrase.value` に一時保持されるのみで、保存処理には乗らない。

### E-5. 二重の安全固定（main.js:851）


## 付録F：状態遷移とステータス語彙

`CommandRequest.Status` が取り得る値と遷移：

```
              ┌──────────── reject ───────────┐
              ▼                                │
 (新規要求) → 承認待ち ── approve(成功) ──→ 実行済み
              │  └──── approve(失敗) ──→ 実行失敗
              │
              │  (bypass エージェント)
              └─ 自動実行中 ─ 成功 ─→ 自動実行済み
                            └ 失敗 ─→ 自動実行失敗
              │
              └──── reject ──→ 拒否
```

- 「承認待ち」だけが承認キュー（`renderCommands` の左ペイン）に表示される。それ以外は履歴テーブルにのみ並ぶ。

`Connection.Status`：`Unknown` → 疎通テスト成功で `Online`、失敗で `Error`、明示切断で `Disconnected`。フロントは `Online` を緑、`Error` を赤、それ以外をグレーのバッジで表示する。

`MCPSettings.Status`（日本語）：`停止中` / `起動中` / `再起動中` / `エラー`。`setMCPStatus` と `startMCPServer`/`restartMCPServer` が更新する。

## まとめ

`ssh-gete` は、「AI エージェント × SSH 実行」という危うい組み合わせに **人間の承認関門** を挿入することで安全を担保する、ローカル完結型のデスクトップアプリです。

- バックエンド（Go / `app.go` 中心）が状態の唯一の所有者として、接続先・承認キュー・ポリシー・MCP 設定を `sync.RWMutex` 下で管理し、SQLite に永続化する。
- MCP サーバー（`mcp_server.go`）がエージェントとの JSON-RPC 接点で、コマンドを**原則キューに積むだけ**で実行はしない。シリアル接続先は MCP からは実行不可。
- SSH 層（`ssh_exec.go`）が認証・タイムアウト・出力上限つきで実際の実行を担う。
- ターミナル層（`terminal.go`）が、トランスポート非依存の `terminalSession` で **SSH PTY シェルとシリアルコンソール**を同一の対話 UI（xterm.js）に統合し、Wails イベントで双方向ストリーミングする。
- フロントエンド（`main.js`）はフレームワークなしの素朴な全画面再描画で、承認・接続管理・**ターミナル**・設定の4画面を提供し、編集保護つきで5秒ポーリングする（ターミナル中は再描画を抑止して xterm を守る）。
- テスト（`mcp_server_test.go`）が「Low でも自動実行しない」「エージェント名必須」「シリアルは MCP 拒否」「履歴 ID の一意化」「永続化」といった契約を回帰検証する。

このドキュメントを地図に、各ファイルの該当行（`file:line`）を辿れば、機能追加や挙動変更の影響範囲をすばやく把握できるはずです。
